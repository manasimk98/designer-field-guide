NOTES

Field Notes: Collaborating Well

Habits that matter more than vocabulary.

Hand off the live thing, not just the static thing

Link the running prototype or staging build alongside your Figma file. Engineers can inspect real states, spacing, and behavior far faster than they can guess from a flat screen.

Write repro steps, not vibes

"It's broken on mobile" takes ten minutes to track down. "1. Open on iPhone width. 2. Tap the filter icon. 3. Cards overlap the footer" takes ten seconds.

Ask which environment you're looking at

Before reacting to something looking wrong, confirm it's dev, staging, or production. Half of "this is broken" turns out to be "this hasn't been deployed yet."

Name it bug or scope, not just "issue"

A bug is a deviation from what was agreed. Anything new is a feature request. Saying which one you mean changes how it gets picked up, and how fast.

Comment on the PR, not just in Slack

If a build is linked to a pull request, leaving feedback there keeps it attached to the actual code change, so nothing gets lost once the thread scrolls away.

Match your ask to the scale of the word

"Can we prompt it differently" is a five-minute change. "Can we fine-tune it" is a project. Using the right word signals you understand the size of what you're asking for.

Give AI output an accessibility pass

AI-generated code and mockups routinely skip alt text, color contrast, and keyboard navigation. Before anything ships, check those three by hand. The AI won't flag them for you.

No signal? Ask "which environment am I looking at?" before you react to anything looking broken.

Territory: Git & GitHub, in practice

Setting Up
GitHub

This is the point-and-click path to cloning a project, making a change, and opening your first pull request.

OVERVIEW

How a Change Actually Travels

Before the step-by-step, here's the whole trip a design change takes, from your computer to the live product.

Steps 1 through 3 happen only on your computer. Nothing is shared with anyone until step 4.

STEPS

Setting Up & Making Your First Change

Do this once, then repeat steps 6 through 9 every time you touch a project. All nine steps use GitHub Desktop, a visual app, not a terminal.

1

Start here

Create a GitHub account

Sign up free at github.com with a professional email. This becomes your permanent identity on every project you touch.

2

Install GitHub Desktop

Download the free app from desktop.github.com. It's the point-and-click way to use Git, so you'll almost never need Terminal.

3

Sign in inside the app

Open GitHub Desktop and log in with the account you just made. This links every change you make back to your name.

4

Get the repo URL from your team

Ask an engineer for the link to the specific project, something like github.com/company/project-name.

5

Clone the repository

File → Clone Repository, paste the URL, pick a folder on your computer. This downloads the entire project, with its full history.

6

Safety checkpoint

Create a branch before touching anything

Branch → New Branch, name it something like yourname/nav-tweak. This keeps your changes isolated from the live version until they're reviewed.

7

Make your changes

Edit whatever files you were asked to. If you're only reviewing, you can stop right here. Most designers never go past step 6.

8

Commit your changes

GitHub Desktop shows exactly what changed. Write a short, honest summary and click "Commit to [branch name]."

9

Shipped for review

Push, then open a pull request

Click "Push origin," then "Create Pull Request." It opens right in your browser, ready for an engineer to review.

READING

Anatomy of a Repo Page

What you're looking at the first time an engineer sends you a GitHub link. This is an actual repo page, with the username, name, and profile photos blurred out.

A real repo.

  • 1

    Code / Issues / Pull requests tabs. Switch between browsing files and reviewing work in progress.

  • 2

    Branch switcher. Shows which version you're looking at. Check this before reacting to anything looking wrong.

  • 3

    Code / Clone button. Where you copy the repo URL you need for step 4 of the setup above.

  • 4

    Readme link. Sits in the sidebar under "About." Click it to open the file most teams use to explain what a project is and how to run it.

  • 5

    Watch, Fork, and Star. Watch follows notifications for the repo, Fork copies it under your own account, and Star just bookmarks it. None of these touch the actual code.

  • 6

    Private badge. Shows this repo is only visible to people who've been given access. A public repo just skips this label.

  • 7

    Go to file. A quick search box for jumping straight to a filename instead of scrolling the whole list.

  • 8

    Commit history. Click the commit count to see every saved checkpoint on this branch, oldest to newest.

  • 9

    Branch protection banner. A nudge, not an alarm. It just means force-pushes or deletions to main aren't blocked yet. Worth mentioning to an engineer, not something to panic over.

Stuck on a merge conflict? That's an engineer's job. Just flag it, don't try to resolve it alone.

Territory: The Toolbelt

The AI
Tool Shelf

Some examples of AI tools that designers can use, and what each one is good for.

CAUTION

Before You Paste Anything

Worth reading before you drop something into any tool on this page.

Assume it's not private

Most AI chat tools may use what you paste to improve their models unless you've specifically turned that off in settings. Treat anything you type as something the vendor could see.

Skip client work and proprietary code

If you wouldn't post it in a public forum, don't paste it into a tool you haven't checked the data settings on. This includes screenshots, not just text.

Watch for personal data

Names, emails, and other identifying details in test data or real user content deserve the same caution as anything else confidential.

CHAT

AI Chat & Research

General-purpose assistants for thinking things through, drafting copy, and research.

Claude

Anthropic's assistant, strong at writing, research, and reasoning through ambiguous problems in plain conversation.

Best for: turning a messy brief into a clear plan, or drafting copy and docs.

ChatGPT

OpenAI's general-purpose assistant with the widest plugin ecosystem, plus built-in image generation in the same chat.

Best for: quick research and one-off images without switching apps.

Gemini

Google's assistant, tightly woven into Docs, Sheets, and Meet.

Best for: teams already living inside Google Workspace.

Perplexity

A research assistant built around live web search, so every answer comes with clickable sources you can actually verify.

Best for: fact-checking, competitive research, and anything where you need to show your sources.

DESIGN → CODE

Design-to-Code & Fast Prototyping

The tools actually collapsing the handoff, turning a prompt, a frame, or a screenshot into something real.

Figma Make

Figma's built-in AI that turns a frame, prompt, or attached file into a working, code-backed prototype. You can preview it and ship it as a pull request, without leaving Figma.

Best for: going from a Figma frame to something clickable and real.

Figma First Draft

Generates a wireframe or basic screen from a text prompt inside Figma Design itself, before any code is involved.

Best for: breaking a blank-canvas moment, fast.

Framer AI

Describe a page or full site and it generates a live, animated, publish-ready website you can keep editing visually.

Best for: marketing sites, landing pages, and portfolios that need to look polished immediately.

v0 by Vercel

Describe a UI, or upload a screenshot or Figma frame, and get back clean, production-quality React code.

Best for: real code that drops straight into an existing product, not just a demo.

Claude (Artifacts)

Beyond chat, Claude can generate a working prototype, document, or mini-app right in the conversation that you can preview, edit, and share on its own.

Best for: quick interactive mockups without setting up any tools.

Lovable

A full-stack app builder that wires up a real backend, database, login, the works, along with the interface, not just the frontend.

Best for: a working MVP a founder or PM can demo without an engineer's help.

Bolt.new

A browser-based coding environment that builds full-stack apps from a prompt and lets you see and edit the actual code as it's generated.

Best for: technically curious people who want to peek under the hood while prototyping.

Google Stitch

Google's take on design-to-code. It stays on the design side longer than v0 does, generating UI you can refine before exporting.

Best for: designers who want more visual control before code enters the picture.

DEV TOOLS

What Your Engineers Are Using

These are mostly built for engineers, but you're welcome to explore them too. Knowing they exist explains a lot about how fast things move now.

GitHub Copilot

An AI pair-programmer that lives inside an engineer's code editor, suggesting and writing code as they type. Deeply tied to GitHub's pull request and review workflow.

Why it matters to you: part of why a "quick fix" can turn around in an hour.

Cursor

A full AI-native code editor that can plan and execute changes across many files at once, not just suggest single lines.

Why it matters to you: teams using it tend to move faster on larger refactors.

Claude Code

Anthropic's AI coding agent, working from the terminal, that handles larger, multi-step engineering tasks with less hand-holding.

Why it matters to you: same idea as Copilot and Cursor, and another name you'll hear in standups.

Windsurf

Another AI-native code editor in the same space as Cursor, built by the team formerly behind Codeium.

Why it matters to you: another name in the same category, so don't worry about matching it to a specific engineer's setup.

IMAGES

Image & Asset Generation

For moodboards, placeholder imagery, and exploratory visuals, each with a different strength.

Midjourney

The current benchmark for artistic, high-impact imagery. Style and mood over precise control.

Best for: moodboards, concept art, campaign visuals.

Adobe Firefly

Trained only on licensed content, making it the safest choice when an image needs to survive legal or commercial review. Built into Photoshop and Illustrator.

Best for: anything that needs to be commercially defensible.

Ideogram

The most reliable tool for generating images with actual legible text inside them.

Best for: posters, banners, mocked-up ads.

ChatGPT / GPT Image

A strong all-purpose option when you just need something quick and accurate, in the same window you're already working in.

Best for: fast, low-stakes visuals mid-conversation.

Recraft

Generates vector graphics and icons you can actually edit as vectors, not just flat images.

Best for: logos, icon sets, and anything that needs to scale cleanly.

Check the license before it ships

Before anything goes in front of a client or into production, check that tool's commercial-use terms. Adobe Firefly is currently the only one of these with formal copyright indemnification. The others vary and change often.

MEETINGS

Meeting Notes & Productivity

Worth knowing the "bot vs. bot-free" split. It changes how candid a conversation feels.

Granola

A bot-free notetaker: it listens to your device audio with no visible "recording" bot in the call, and turns your own rough notes into a structured summary afterward.

Best for: client calls and stakeholder conversations.

Otter / Fireflies / Fathom

Bot-based meeting assistants that join the call, transcribe everything, and generate searchable summaries and action items.

Best for: internal team syncs where everyone's comfortable being recorded.

Tools change fast in this space. Treat this list as a starting point, not gospel.

A different skill from using AI

Designing
AI Features

The AI Tools page covers using AI to build faster. This page covers designing products that have AI built into them, which is a different job entirely.

PATTERNS

Core Interaction Patterns

The building blocks that show up in almost every AI-powered feature, whether anyone notices them or not.

Streaming response

Show text as it generates instead of a blank screen followed by a wall of text.

Best for: anything that takes more than a second or two. People tolerate a slower start far better than a long silent wait.

Thinking / processing indicator

A visible state that shows the system is working on something more involved than a simple lookup.

Different from a generic spinner. It sets the expectation that this step takes real work.

Escape hatch

A clear way to stop, edit, or override an AI action mid-flight.

Especially important right before anything irreversible happens, like sending a message or deleting a file.

Regenerate or retry

A one-click way to ask for another attempt at the same request.

A single AI response is rarely treated as final. Design for the retry, not just the first try.

TRUST

Handling Uncertainty and Errors

AI will sometimes be confidently wrong. The interface has to plan for that, not just the happy path.

Graceful hallucination handling

Since AI can state something wrong with total confidence, design as if any answer could be off.

Cite sources where possible, flag lower-confidence answers, and make corrections easy instead of treating every output as final.

Confidence signals

A way to show how sure the system actually is, instead of presenting every answer with the same authority.

A citation, a source link, or a simple "double-check this" nudge all do this job.

Empty and fallback states

What the interface shows when the AI genuinely has nothing good to offer.

A dead end is worse than an honest "I'm not sure." Design this state on purpose, don't let it happen by accident.

CONTROL

Feedback and Control

AI systems feel less like a black box when people have a way to react to and adjust what comes out of them.

Thumbs up or down

A lightweight way for people to signal whether a response was useful.

Improves the product over time and gives people a sense of agency over a system that otherwise feels opaque.

Human in the loop

Keeping a person confirming any consequential action, like sending an email, deleting something, or spending money.

Don't let AI carry out irreversible steps fully on its own.

Editable AI output

Treating AI-generated content as a draft, not a final answer.

Wherever someone might reasonably want to adjust the result, let them edit it directly instead of only offering regenerate.

HONESTY

Transparency and Disclosure

People trust a system more, not less, when they can see roughly what it's doing and why.

Label AI-generated content

Making it clear, through a badge, a caption, or a tone shift, when something was generated rather than authored.

Especially important for images, summaries, and written copy that could pass as human-made.

Explain the reasoning

Showing roughly why an answer was given, instead of presenting it with no context at all.

Even something simple, like "based on your last 3 orders," builds more trust than a bare answer.

NOTES

Before You Ship

Habits that catch the gaps a happy-path demo won't.

Test with a wrong answer, not just a right one

Every AI demo looks great with a clean example. Deliberately try to break it, ask something ambiguous or unanswerable, and see what the interface actually shows.

Give every AI action an undo

If a person can't undo it, don't let AI do it without a confirmation step first.

Don't hide the label to seem more impressive

If people would feel differently knowing a human didn't make it, they deserve to know.

Design the empty state on purpose

Decide what happens when the AI has nothing good to say, rather than letting whatever the model outputs become the default.

Every AI feature eventually meets a case nobody tested for. Design like you already know that.
